# One Blood Story Map

One Blood Story Map is a React + Leaflet + Supabase app for sharing hometown stories across a global map.

## Setup

1. Install dependencies:
   ```bash
   npm install
   ```
2. Create `.env` from `.env.example` and fill:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
3. Start development server:
   ```bash
   npm run dev
   ```

## App routes

- `/` Landing page
- `/map` Story map and browsing panel
- `/login` Login page
- `/signup` Sign up page
- `/forgot-password` Request password reset email
- `/reset-password` Set new password from recovery email link
- `/verify-email` Email verification instructions page
- `/auth/callback` Supabase auth callback route (handled by verification page)

## Supabase table

If this is a new project, run:

```sql
create table if not exists public.stories (
  id bigint generated by default as identity primary key,
  user_id uuid references auth.users(id),
  name text,
  pronouns text,
  hometown text not null,
  country_code text,
  story text,
  share_text text,
  audio_url text,
  story_type text check (story_type in ('text', 'audio', 'both', 'profile')),
  graduation_year integer,
  social_links jsonb not null default '[]'::jsonb,
  latitude double precision not null,
  longitude double precision not null,
  created_at timestamptz not null default now()
);
```

  ) then

If you want to require at least one of text or audio, add this optional constraint:

```sql
alter table public.stories
  add constraint stories_content_check
  check (
    char_length(trim(coalesce(story, ''))) > 0
    or char_length(trim(coalesce(audio_url, ''))) > 0
  ) not valid;
```

## Auth + access control setup

### 1. Enable email confirmation

In Supabase Auth settings, ensure email confirmation is required.

### 1.1 Configure password reset redirect URL

In Supabase Auth URL configuration, add this redirect URL:

- `https://your-domain/reset-password` (production)
- `http://localhost:5173/reset-password` (local development)

### 2. Enforce Berea domain at DB policy level

The app validates domain on sign-up in UI (`@berea.edu` by default), and these RLS policies enforce it server-side too.

```sql
alter table public.stories enable row level security;

create policy "Stories are public to read"
on public.stories
for select
to public
using (true);

create policy "Verified Berea users can insert stories"
on public.stories
for insert
to authenticated
with check (
  auth.uid() = user_id
  and auth.jwt() ->> 'email' ilike '%@berea.edu'
);

create policy "Verified Berea users can update own story profile"
on public.stories
for update
to authenticated
using (
  auth.uid() = user_id
  and auth.jwt() ->> 'email' ilike '%@berea.edu'
)
with check (
  auth.uid() = user_id
  and auth.jwt() ->> 'email' ilike '%@berea.edu'
);
```

Notes:

- If you need to support another community domain later, update both the frontend constant (`src/auth/authConfig.js`) and the policy.
- Email verification is handled by Supabase Auth email confirmation flow.

## Supabase storage setup

1. In Supabase dashboard, go to `Storage` and create a public bucket named `story-audio`.
2. If you use RLS for storage, allow reads and inserts for this bucket.

```sql
insert into storage.buckets (id, name, public)
values ('story-audio', 'story-audio', true)
on conflict (id) do nothing;

create policy "Public can read story audio"
on storage.objects
for select
to public
using (bucket_id = 'story-audio');

create policy "Authenticated can upload story audio"
on storage.objects
for insert
to authenticated
with check (
  bucket_id = 'story-audio'
  and auth.jwt() ->> 'email' ilike '%@berea.edu'
);
```
